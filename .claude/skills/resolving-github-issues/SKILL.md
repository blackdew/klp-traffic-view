# Resolving GitHub Issues (end-to-end)

여러 GitHub 이슈를 "확인 → 작업 → 검증 → 댓글 → 닫기"까지 한 흐름으로 처리한다.
목표는 단순히 코드를 바꾸는 게 아니라, **각 이슈를 닫아도 되는 근거를 남기는 것**이다.

## 핵심 원칙 (왜 이렇게 하나)

- **분석 먼저, 구현 나중**: 코드를 읽고 기존 패턴·의존성을 파악하기 전에 손대면, 컨벤션을 깨거나 잘못된 순서로 작업해 되돌리게 된다.
- **검증 없이 완료 보고 금지**: "될 것 같다"와 "된다"는 다르다. 실제로 실행해 눈으로 확인한 것만 완료로 본다.
- **댓글은 추적 가능성**: 커밋 SHA를 댓글에 적으면 GitHub이 커밋↔이슈를 양방향 링크한다. 검증 근거를 함께 남기면 "왜 닫았는가"가 이슈의 완료 기준과 1:1로 대응된다.

## 워크플로우 체크리스트

작업 시작 시 이 체크리스트를 응답에 복사해 진행 상황을 추적한다:

```
이슈 처리 진행:
- [ ] 1. 이슈 확인 — gh issue list + 각 이슈 본문(요구사항·완료 기준) 읽기
- [ ] 2. 의존성·순서 분석 — 이슈 간 선행 관계·공유 파일 파악
- [ ] 3. 코드 분석 — 관련 코드·패턴·git 이력 파악 (구현 전)
- [ ] 4. 구현 — 컨벤션을 따르는 최소 변경
- [ ] 5. 검증 — 실제 실행/관찰로 동작 확인
- [ ] 6. 커밋 — 이슈 번호 참조, 검증 근거 포함
- [ ] 7. 댓글 + 닫기 — 작업 요약·검증 근거·SHA 댓글 후 completed로 닫기
```

### 1. 이슈 확인

`gh issue list`로 대상을 파악하고, 각 이슈의 **본문과 완료 기준(acceptance criteria)**을 읽는다.
완료 기준이 없으면, 무엇을 만족해야 "닫아도 되는지"를 먼저 정의한다 — 이게 6·7단계 검증의 기준이 된다.

### 2. 의존성·순서 분석

이슈 간 **선행 관계**(예: "Pages 활성화"는 "index.html 생성" 이후)와 **같은 파일을 건드리는 이슈**를 찾는다.

- 분석은 병렬로 해도 되지만, **같은 파일을 수정하는 코드 변경은 순차**로 한다 — 동시에 고치면 충돌한다.
- 선행 관계가 있으면 위상 순서대로 처리한다. 독립적인 이슈는 묶어서 한 번에 구현해도 된다.

### 3. 코드 분석 (구현 전)

관련 코드를 읽어 **기존 패턴·네이밍·구조**를 파악한다. 버그 수정이면 `git log`·`git blame`으로 변경 이력과 회귀 여부를 먼저 본다.
"이 변경이 다른 코드에 어떤 영향을 주는가"를 답할 수 있어야 구현에 들어간다.

### 4. 구현

기존 코드와 **같은 스타일로** 최소 변경한다. 한 이슈가 여러 파일에 걸치면, 논리적으로 묶어 작업한다.
새 기능이라면 주변 컨벤션(주석 밀도·네이밍·관용구)을 그대로 따른다.

### 5. 검증 — 무엇이 "동작"인가

**실제로 실행해 관찰한다.** 테스트 통과 / 앱 렌더 / HTTP 200 / 기대한 출력처럼 객관적 근거를 확보한다.

검증의 함정 하나: **에러 ≠ 무조건 실패**. 콘솔 에러나 실패 응답이 보이면, 그게 *의도된 동작*인지 먼저 확인한다.
예: placeholder API 키가 401을 받아 시뮬레이션 모드로 폴백하는 것은 버그가 아니라 설계된 graceful degradation이다.
검증할 수 없는 환경(GPU 필요 등)이면 그 사실을 **명시적으로** 알린다 — 검증한 척하지 않는다.

각 검증 결과를 이슈의 완료 기준과 대응시킨다. 기준을 못 채우면 그 이슈는 아직 닫지 않는다.

### 6. 커밋

이슈 번호를 참조하고, 무엇을 왜 바꿨는지와 검증 결과를 한 줄 담는다. 논리 단위(또는 Phase)로 묶어 커밋한다.
커밋 메시지·본문은 저장소의 기존 언어/형식을 따른다(한국어 저장소면 한국어).

### 7. 댓글 + 닫기

각 이슈에 **작업 요약 + 검증 근거 + 커밋 SHA**를 댓글로 남기고 `gh issue close -r completed`로 닫는다.

```bash
gh issue comment <번호> -F <본문파일>
gh issue close <번호> -r completed
```

**댓글 템플릿:**

```
✅ 완료 (커밋 <sha>)

- <무엇을 어떻게 바꿨는지 요약 — 1~3줄>

검증: <실제로 확인한 근거 — 예: 라이브 URL 렌더 정상, HTTP 200, 테스트 N개 통과>
```

**예시:**

```
✅ 완료 (커밋 f7a0a4d)

- 루트 index.html 생성 — 소스 버전(traffic_view/*.jsx)을 상대경로로 로드, 깔끔한 URL 제공.

검증: 로컬 정적 서버 + 라이브 URL 양쪽에서 검색 화면 정상 렌더, jsx 페치 200 확인.
```

## 마무리

전부 닫은 뒤, 처리한 이슈 목록과 각 커밋 SHA를 한 번에 요약해 사용자에게 보고한다.
이슈에 연동된 작업 명세(PRD 등)가 있으면 완료 기준 체크박스도 실제 상태에 맞게 갱신한다.

## 이 스킬을 쓰지 않을 때

- 코드 변경 없이 **분석·계획만** 원할 때 → 일반 분석으로 처리.
- 여러 이슈를 **병렬 멀티에이전트로 동시에** 돌리고 싶을 때 → `work-issue-agents`.
- 닫을 이슈가 아니라 **새 이슈를 등록**만 할 때 → `gh issue create` 직접 사용.
